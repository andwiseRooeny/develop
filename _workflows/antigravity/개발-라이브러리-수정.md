**[Header]**: [개발-공통-준수-규칙](./개발-공통-준수-규칙.md)

### 실행 단계

0. **[헤더 참조 및 세계관 동기화] (CRITICAL)**
   - 워크플로우 상단에 정의된 모든 **Header(참조 문서)**를 식별하고, 해당 파일들을 반드시 `view_file` 도구로 열어 내용을 확인합니다.
   - 각 문서에 정의된 규칙, 명명법, 분류 체계 등의 **세계관(Worldview)**을 완벽히 숙지하고 이를 절대적으로 준수하여 업무를 수행합니다.


0. **[프로젝트 루트 확인]**
   - 실행 전 반드시 `_develop/_info/Project_Root.md` 파일이 존재하는지 확인합니다.
   - 프로젝트 루트를 기준으로 모든 리소스 경로를 상대 경로로 계산합니다.

1. **[대상 및 소스 식별]**
   - **수정 대상**: `_develop/_lib/` 하위의 특정 라이브러리 파일.
   - **분석 소스**: 추가할 정보가 담긴 소스 코드, 로그, 또는 사용자의 직접 입력 텍스트.

2. **[통합 분석 (Integrated Analysis)]**
   - **맥락 로드**: 기존 라이브러리 내용과 신규 소스를 하나의 시스템으로 통합 분석합니다.
   - **종합 요약**: 현재 라이브러리의 역할과 이번 수정으로 얻게 될 기술적 가치를 3~4줄로 요약 제안합니다.
   - **보강 포인트 추출**: 기존 해결 방안 중 수정/확장이 필요한 부분과 새롭게 추가될 기술 노하우를 정의합니다.

3. **[수정 옵션 제안]**
   - 분석 결과에 따라 다음 항목을 포함한 최대 6가지 수정 옵션을 제안합니다.
     - [해결 방안 구체화], [참조 자산(Referenced Assets) 업데이트], [실전 회고 및 주의사항 추가] 등
   - 사용자의 번호 선택에 따라 문서 수정을 준비합니다.

4. **[라이브러리 업데이트 실행]**
   - 선택된 옵션을 반영하여 문서를 업데이트합니다.
   - **작성 규칙**:
     - **절대 경로 사용 금지**: 모든 링크는 `./` 또는 `../` 기반의 상대 경로를 사용합니다.
     - **선언적 기술**: 장황한 설명 대신 핵심 선언문과 참조 코드 블록 위주로 구성합니다.
     - **근거 명시**: 수집된 분석 데이터(`_ref` 등)에 기반하여 내용을 보강합니다.

5. **[결과 보고]**
   - 수정된 라이브러리 파일을 사용자에게 링크로 보여주며 작업을 마무리합니다.
